# Project Instructions

## Language

- 默认使用中文与用户沟通。
- 说明尽量简洁，优先给出Plan让我选择之后再执行。

## Working Style

- 先快速理解当前目录和相关文件，再开始修改。
- 能直接落地的任务就直接执行，不只停留在分析或方案。
- 不要改动与当前任务无关的文件。
- 改代码前，先用一句话说明准备修改什么。
- 写代码之后写好简洁易懂的注释

## Execution Loop

- 对于已确认的多步计划，默认持续推进，不需要每一步都等待用户在线确认。
- 采用固定闭环：主线程产出设计/计划/实现，reviewer 子线程按明确 rubric 评审。
- reviewer 只输出：`通过/不通过`、最多 `3` 条关键问题、以及必要的保留点。
- 若 reviewer 不通过，只修明确问题，不无限返工；同一阶段默认最多进行 `3` 轮评审。
- 若 reviewer 通过，立即进入下一阶段继续推进。
- 用户离线时继续工作，直到：
  - 遇到真实 blocker
  - 出现高风险且不可合理假设的决策
  - 当前阶段已完成并通过 reviewer
- 修改实现时优先走 `TDD + 关键路径验证`，完成一个阶段后再进入下一阶段。

## Model Strategy

- 小任务优先使用小模型，降低延迟和成本。
- 小任务包括：单文件小改动、文案修改、配置微调、简单排错、只读检查。
- 这类任务优先选择 `gpt-5.4-mini` 或同级轻量模型，并使用较低到中等推理强度。
- 大任务、复杂重构、跨文件实现、关键缺陷修复、架构设计、疑难问题排查，使用顶配策略。
- 这类任务优先选择 `gpt-5.4`，并将 `reasoning_effort` 提升到 `high` 或 `xhigh`。
- 如果任务风险高、影响面大或需求不清晰，先补上下文再决定是否切换到顶配模型。

## Execution Preference

- 优先采用最小改动完成目标。
- 先验证关键路径，再补充必要说明。
- 若存在多种实现方案，优先选择稳定、可维护、易验证的方案。

## Architecture

- 设计和实现时优先考虑中长期演进，不只修眼前问题。
- 默认采用模块化、可扩展、低耦合的结构：`runtime / skill / tool / knowledge / policy` 尽量解耦。
- 新能力优先做成可替换模块或适配层，避免把 provider、业务规则或页面流程写死在单个文件里。
- 当需求暂时只覆盖单一场景时，实现上仍尽量保留向多领域、多模型、多工具扩展的接口边界。
